# المرحلة الثانية: أونبوردنج التاجر بعد OTP

بعد OTP، إذا كان حساب التاجر جديدًا ولم يكتمل `onboardingCompleted`، يجب أن تظهر شاشة أونبوردنج قبل الدخول الكامل للوحة التحكم.

الأونبوردنج الحالي يركز على بيانات أساسية مثل:

1. هوية المتجر.
2. نشاط العمل.
3. معلومات التواصل.

## 11. خطوات التنفيذ

1. بعد نجاح OTP، راقب هل تظهر شاشة أونبوردنج.
2. في خطوة هوية المتجر، أدخل أو راجع:
   - اسم المتجر بالعربي: برو سبورت اليمن.
   - اسم المتجر بالإنجليزي: Pro Sport Yemen.
   - وصف مختصر إن ظهر الحقل.
3. في خطوة نشاط العمل، اختر:
   - `fashion` إذا كان التركيز على الملابس والأحذية.
   - `other` إذا كان التركيز على الأدوات والكرات والأوزان.
4. سجل هل غياب فئة Sports كان مربكًا.
5. في خطوة معلومات التواصل، أدخل رقم الهاتف أو البيانات المطلوبة.
6. أكمل الأونبوردنج.
7. تأكد أن النظام انتقل إلى لوحة التاجر الكاملة.
8. أعد تحميل الصفحة وتأكد أن الأونبوردنج لا يظهر مرة أخرى لنفس الحساب.

# المرحلة العاشرة: إضافة المنتجات

## 32. قواعد عامة

لكل منتج أساسي، اختبر الحقول التالية عند توفرها:

- اسم عربي وإنجليزي.
- وصف عربي وإنجليزي.
- تصنيف واحد أو أكثر.
- Brand.
- نوع المنتج: `single` للمنتجات الأساسية.
- المنتجات `bundled` و `digital` اختيارية فقط ولا توقف اعتماد السيناريو.
- الحالة والظهور.
- السعر.
- سعر المقارنة `compareAtPrice` إذا كان هناك خصم ظاهر.
- خصم مدمج `inlineDiscount` إذا كان متاحًا، بنسبة أو مبلغ ثابت مع تاريخ بداية ونهاية.
- تكلفة المنتج `costPrice` إن ظهرت.
- المخزون أو مخزون غير محدود.
- حد المخزون المنخفض.
- SKU و barcode.
- وزن وأبعاد، خصوصًا للكرات والأوزان.
- حد أدنى وأقصى للطلب.
- وسوم Tags مثل: رياضة، رجالي، تمرين، كرة قدم، جري.
- تسمية منتج Product Label مثل: جديد، الأكثر طلبًا، مناسب للجيم.
- رابط YouTube لفيديو استخدام إن وجد.
- SEO title و SEO description.
- منتجات مرتبطة.
- أسئلة المنتج مفعلة إن ظهر الخيار.
- الضرائب `isTaxable` و `taxRate` إن ظهرت.
- الصور وترتيبها واقتصاصها إن توفر.

# المرحلة الحادية عشرة: طرق الدفع المطابقة للنظام

## 35. أنواع الدفع المدعومة

النظام يدعم أنواعًا تقنية مثل:

| النوع التقني | الاسم المقترح للعرض | ملاحظات مهمة |
|---|---|---|
| cod | الدفع عند الاستلام | مناسب للطلبات المحلية |
| wallet | محفظة / طريقة دفع محلية | مثل كريمي أو جوالي إذا كانت مفعلة من المنصة |
| bank_transfer | حوالة بنكية | تستخدم بيانات حساب بنكي |
| exchange_transfer | حوالة عبر صراف | طريقة صرافة عامة وليست شرطًا أن تكون كريمي |
| custom_manual | طريقة يدوية مخصصة | لا تختبرها إلا إذا كانت مضافة من Platform Admin |

> ملاحظة مهمة: كريمي في الإعدادات الافتراضية قد يظهر كطريقة دفع محلية من نوع `wallet` وليس كـ `exchange_transfer`. لذلك لا تفترض أن كريمي يجب أن يكون حوالة صرافة.

## 36. حدود صلاحيات التاجر في طرق الدفع

في لوحة التاجر، المختبر يختبر غالبًا:

- تفعيل/تعطيل طريقة دفع موجودة.
- تعديل اسم الحساب.
- تعديل رقم الحساب أو رقم المحفظة.
- تعديل رقم الهاتف.
- تعديل IBAN إن كان موجودًا.
- كتابة تعليمات الدفع بالعربي/الإنجليزي إن ظهرت.
- ترتيب الطرق إن ظهر.

لكن خصائص مثل:

- يتطلب مرجع.
- يتطلب إيصال.
- الإيصال اختياري أو إجباري.

قد تكون مضبوطة على مستوى المنصة، وليست من صلاحيات التاجر في لوحة التاجر.

## 37. طرق الدفع المطلوبة في الاختبار

### الدفع عند الاستلام COD

1. فعّل طريقة COD.
2. تأكد أنها تظهر للعميل في checkout.
3. نفذ طلبًا بها.
4. تحقق أن checkout لا يطلب إيصالًا أو مرجعًا لهذه الطريقة إلا إذا ظهر خلاف ذلك.

### كريمي أو محفظة محلية

اختبر طريقة كريمي أو أي محفظة محلية إذا كانت مفعلة.

| الحقل | القيمة |
|---|---|
| اسم الحساب | Pro Sport Yemen |
| رقم الحساب/المحفظة | 123456789 |
| رقم الهاتف | +9677xxxxxxx |
| التعليمات بالعربي | حول المبلغ ثم اكتب رقم العملية وارفع الإيصال إن طلبته الصفحة. |
| التعليمات بالإنجليزي | Transfer the amount, enter the reference number, and upload receipt if requested. |

### حوالة بنكية

| الحقل | القيمة |
|---|---|
| اسم الحساب | Pro Sport Yemen |
| رقم الحساب | 000111222333 |
| IBAN | YE00TEST000111222333 |
| التعليمات | ارفع إيصال التحويل بعد الدفع إذا طلبته الصفحة. |

### حوالة عبر صراف عامة

اختبر `exchange_transfer` إذا ظهرت كطريقة مستقلة.

| الحقل | القيمة |
|---|---|
| اسم المستلم | Pro Sport Yemen |
| رقم الهاتف | +9677xxxxxxx |
| التعليمات | أرسل الحوالة عبر شركة صرافة ثم أدخل رقم العملية. |

## 38. اختبار الإيصال والمرجع

1. اختر طريقة دفع تظهر أنها تطلب رقم مرجع.
2. جرّب إرسال الطلب بدون رقم مرجع.
3. المتوقع: إذا كانت الطريقة تتطلب مرجعًا، يجب منع الإرسال برسالة واضحة.
4. اختر طريقة دفع يظهر فيها رفع الإيصال.
5. ارفع إيصالًا تجريبيًا.
6. جرّب أيضًا إرسال الطلب بدون إيصال إذا كان مكتوبًا أنه اختياري.
7. إذا كانت الصفحة تقول إن الإيصال إجباري ثم سمحت بالإرسال بدونه، سجلها مشكلة عالية.

## 39. تحقق المرحلة

- [ ] وجدت صفحة طرق الدفع.
- [ ] فهمت الفرق بين COD والمحفظة والتحويل البنكي وحوالة الصرافة.
- [ ] استطعت تفعيل/تعطيل طريقة دفع.
- [ ] استطعت إدخال بيانات الحساب.
- [ ] استطعت إدخال تعليمات عربي/إنجليزي إن ظهرت.
- [ ] ظهرت الطريقة في صفحة الدفع للعميل.
- [ ] ظهر حقل المرجع فقط عند الحاجة.
- [ ] ظهر رفع الإيصال عند الحاجة أو كاختياري.
- [ ] رسائل الخطأ واضحة عند نقص المرجع أو الإيصال إذا كان إجباريًا.

## 40. اختبار مهم قبل إعداد التوصيل

قبل إنشاء أي منطقة أو طريقة توصيل، افتح checkout بمنتج في السلة.

سجل النتيجة:

- هل يمنع النظام إكمال الطلب برسالة واضحة؟
- هل يخبر العميل أنه لا توجد طرق توصيل متاحة؟
- هل يوجه التاجر لإعداد التوصيل؟
- هل يسمح بإرسال طلب بدون طريقة توصيل؟

إذا سمح بإكمال طلب يحتاج توصيل بدون طريقة توصيل واضحة، سجلها مشكلة عالية.

## 41. مناطق التوصيل المطلوبة

| الاسم | المدينة/المحافظة | المنطقة | الرسوم الافتراضية | الحالة |
|---|---|---|---|---|
| صنعاء داخل المدينة | أمانة العاصمة | داخل المدينة | 1500 | فعالة |
| صنعاء أطراف | صنعاء | أطراف المدينة | 2500 | فعالة |
| عدن | عدن | داخل المدينة | 3500 | فعالة |
| شحن منتجات ثقيلة | أمانة العاصمة | أدوات ثقيلة | 5000 | فعالة |

## 42. طرق الشحن/التوصيل المطلوبة

اختبر ما هو متاح في الواجهة:

| النوع | اسم العرض | الحقول المهمة |
|---|---|---|
| flat_rate | توصيل ثابت داخل صنعاء | cost، minDeliveryDays، maxDeliveryDays |
| by_weight | توصيل حسب الوزن | baseCost، costPerKg، minCost، maxCost |
| by_item | توصيل حسب عدد القطع | baseCost، costPerItem |
| free_shipping | شحن مجاني للطلبات الكبيرة | freeShippingConditionType، freeShippingMinOrderAmount |
| store_pickup | استلام من المتجر | pickupCost، أيام التوصيل |
| weight_tier | شرائح حسب الوزن | ranges |
| order_value_tier | شرائح حسب قيمة الطلب | ranges |

## 43. اختبار الاستلام من المتجر

نفذ طلبًا مستقلًا أو محاولة checkout باستخدام `store_pickup` إن كان متاحًا.

تحقق من:

- هل يظهر خيار الاستلام من المتجر بوضوح؟
- هل يقلل الحاجة إلى عنوان توصيل كامل؟
- هل تظهر تكلفة الاستلام إن وجدت؟
- هل يظهر في تفاصيل الطلب أنه استلام من المتجر؟

## 44. تحقق المرحلة

- [ ] استطعت إنشاء منطقة توصيل.
- [ ] استطعت إنشاء أكثر من طريقة توصيل/شحن.
- [ ] فهمت معنى نوع كل طريقة.
- [ ] ظهرت طرق التوصيل في checkout.
- [ ] تغيرت الرسوم حسب المنطقة أو الطريقة.
- [ ] المنتجات الثقيلة يمكن اختبارها بطريقة حسب الوزن أو تكلفة أعلى.
- [ ] ظهر خيار الاستلام من المتجر إن تم تفعيله.
- [ ] ظهر الشحن المجاني فقط عند تحقق الشرط.
- [ ] checkout بدون طرق توصيل أعطى رسالة مفهومة.

# المرحلة الثالثة عشرة: تجربة العميل في المتجر

## 45. التصفح والبحث

1. افتح رابط المتجر العام.
2. تأكد من ظهور الهوية والصور بشكل مناسب للرياضة والملابس الرجالية.
3. افتح صفحة المنتجات.
4. اختبر البحث عن:
   - حذاء.
   - تيشيرت.
   - كرة.
   - دمبل.
5. اختبر التصنيفات.
6. اختبر الفلاتر حسب السمات إن ظهرت.
7. افتح صفحة تيشيرت بمتغيرات.
8. اختبر تغيير المقاس واللون.
9. افتح صفحة الحذاء واختبر مقاس الحذاء.
10. افتح صفحة الدمبل واختبر الوزن.
11. افتح المنتج النافد.
12. تأكد أن المنتج النافد لا يسمح بالشراء أو يظهر بوضوح.

## 46. اختبارات تحتاج عميلًا مسجلًا

هذه الاختبارات لا تنفذ كضيف إلا إذا سمح النظام بذلك:

1. تسجيل أو دخول عميل.
2. إضافة منتج للمفضلة.
3. إضافة سؤال على منتج مثل: هل المقاس مطابق للقياس الأوروبي؟
4. إضافة مراجعة إن كان متاحًا.
5. إذا كنت ضيفًا وحاولت هذه العمليات، تحقق أن النظام يطلب تسجيل الدخول برسالة واضحة.

# المرحلة الخامسة عشرة: تتبع الطلب من جانب العميل

## 50. المطلوب الفعلي حاليًا

صفحة تتبع الطلب في النظام الحالي قد تكون مختصرة. لا تفترض أنها تعرض كل تفاصيل الطلب.

المتوقع الأساسي الذي يجب اختباره:

- رقم الطلب.
- حالة الطلب.
- الإجمالي.
- العملة.
- آخر تحديث.
- خط سير الحالة / Timeline إن ظهر.

أما عرض المنتجات، المتغيرات، الدفع، الشحن، والعنوان داخل صفحة التتبع فهو تحسين مطلوب إذا لم يكن موجودًا، وليس شرطًا لنجاح المرحلة الحالية.

## 51. خطوات التنفيذ

1. افتح صفحة تتبع الطلب.
2. أدخل رقم الطلب أو البيانات المطلوبة.
3. تحقق من ظهور:
   - رقم الطلب.
   - حالة الطلب.
   - الإجمالي.
   - العملة.
   - آخر تحديث.
   - Timeline إن ظهر.
4. بعد تغيير حالة الطلب من لوحة التاجر، أعد فتح صفحة التتبع وتأكد من تحديث الحالة.
5. راقب أسماء الحالات في صفحة التتبع.

## 52. اختبار مهم لحالات التتبع

الحالات الفعلية في الباك إند هي:

| الحالة التقنية | التسمية العربية |
|---|---|
| new | جديد |
| confirmed | مؤكد |
| preparing | قيد التجهيز |
| out_for_delivery | في الطريق |
| completed | مكتمل |
| cancelled | ملغي |
| returned | مسترجع |

إذا ظهرت في صفحة التتبع تسميات قديمة مثل:

- pending
- processing
- shipped
- delivered

أو ما يقابلها بشكل غير مطابق، سجلها كمشكلة توافق في الواجهة.

## 53. تحقق المرحلة

- [ ] وجدت صفحة تتبع الطلب.
- [ ] عرفت ما البيانات المطلوبة للتتبع.
- [ ] ظهرت حالة الطلب.
- [ ] ظهر الإجمالي والعملة.
- [ ] ظهر آخر تحديث.
- [ ] تغيرت الحالة بعد تحديث التاجر.
- [ ] تسميات الحالات مطابقة أو سجلت مشكلة.
- [ ] نقص تفاصيل المنتجات والدفع والشحن سجلته كتحسين إن لم تكن ظاهرة.

# المرحلة السابعة عشرة: تغيير حالة الطلب

## 56. الحالات الفعلية

| الحالة التقنية | التسمية العربية |
|---|---|
| new | جديد |
| confirmed | مؤكد |
| preparing | قيد التجهيز |
| out_for_delivery | في الطريق |
| completed | مكتمل |
| cancelled | ملغي |
| returned | مسترجع |

## 57. الانتقالات المسموحة

| من | إلى |
|---|---|
| new | confirmed أو cancelled |
| confirmed | preparing أو cancelled |
| preparing | out_for_delivery أو cancelled |
| out_for_delivery | completed أو returned |
| completed | لا يوجد انتقال |
| cancelled | لا يوجد انتقال |
| returned | لا يوجد انتقال |

## 58. خطوات التنفيذ

1. غيّر الطلب من `new` إلى `confirmed`.
2. أضف ملاحظة إن ظهر حقل الملاحظة.
3. غيّره إلى `preparing`.
4. غيّره إلى `out_for_delivery`.
5. افتح صفحة تتبع الطلب وتأكد من ظهور الحالة.
6. غيّره إلى `completed`.
7. أنشئ طلبًا ثانيًا أو استخدم طلب اختبار آخر لتجربة الإلغاء.
8. جرّب انتقالًا غير مسموح مثل `new` إلى `completed` إن كانت الواجهة تسمح بمحاولة ذلك، وسجل هل تم منعه.

## 59. تحقق المرحلة

- [ ] الحالات المعروضة مطابقة للقائمة الفعلية.
- [ ] لا توجد حالة "قيد المراجعة" كحالة طلب.
- [ ] الانتقالات غير المسموحة ممنوعة أو غير معروضة.
- [ ] الملاحظات تحفظ إن وجدت.
- [ ] حالة العميل في صفحة التتبع تتحدث.

# المرحلة الثامنة عشرة: الإشعارات

## 60. ما يجب فحصه

1. إشعار إنشاء طلب جديد.
2. إشعار مخزون منخفض عند وصول منتج إلى حد المخزون المنخفض.
3. إشعار نفاد المخزون عند وصول المخزون إلى 0.
4. مركز الإشعارات في لوحة التاجر.
5. قائمة الإشعارات في الشريط العلوي.
6. صوت الإشعار إن كان مفعّلًا في الواجهة.
7. زر المزيد ينقل إلى صفحة الإشعارات إن وجد.
8. تفضيلات الإشعارات إن كانت ظاهرة فقط:
   - فورية.
   - ملخص يومي.
   - كتم.
9. التحديث الفوري إن وصل بدون تحديث الصفحة.

أحداث مهمة يجب تسجيلها إن ظهرت:

- `order.created`
- `inventory.low_stock`
- `inventory.out_of_stock`

## 61. تحقق المرحلة

- [ ] ظهر إشعار عند إنشاء الطلب.
- [ ] ظهر الطلب في مركز الإشعارات.
- [ ] ظهرت آخر الإشعارات في الشريط العلوي.
- [ ] زر المزيد يعمل إن وجد.
- [ ] صوت الإشعار يعمل أو سجلت أنه لا يعمل.
- [ ] يمكن تمييز الإشعار كمقروء.
- [ ] تفضيلات الإشعارات واضحة إن ظهرت.
- [ ] الإشعارات الفورية تعمل أو لا تعمل وسجلت النتيجة.
