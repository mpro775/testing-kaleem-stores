# المرحلة الحادية عشرة: طرق الدفع المطابقة للنظام

## 37. ملاحظات مهمة قبل الاختبار

طرق الدفع في النظام الحالي تأتي من إعدادات المنصة، والتاجر غالبًا لا ينشئ نوع دفع جديد من الصفر، بل يفعّل ويعدّل بيانات الطرق المتاحة له.

مهم جدًا:

- `alkuraimi` أو طرق المحافظ المحلية قد تظهر كنوع `wallet` وليس `exchange_transfer`.
- `exchange_transfer` تعني حوالة صرافة عامة، وليست بالضرورة كريمي تحديدًا.
- `custom_manual` نوع مدعوم تقنيًا، لكنه لا يكون متاحًا للتاجر إلا إذا أضافته المنصة مسبقًا.
- حقول مثل “يتطلب مرجع” و “يتطلب إيصال” ليست بالضرورة قابلة للتعديل من لوحة التاجر، وقد تكون محددة من منصة الإدارة.
- الإيصال قد يكون اختياريًا أو إجباريًا حسب إعداد طريقة الدفع في المنصة. لا تفترض أنه إجباري دائمًا.

## 38. أنواع الدفع التي قد تظهر

| النوع التقني | الاسم المقترح للعرض | ملاحظات الاختبار |
|---|---|---|
| cod | الدفع عند الاستلام | لا يحتاج مرجعًا أو إيصالًا غالبًا |
| wallet | محفظة / كريمي / جوالي / ون كاش | كريمي قد يظهر هنا كطريقة محلية |
| bank_transfer | حوالة بنكية | قد تتطلب مرجعًا وقد تسمح بإيصال |
| exchange_transfer | حوالة عبر صراف | صرافة عامة وليست كريمي بالضرورة |
| custom_manual | طريقة يدوية مخصصة | اختبرها فقط إذا كانت مضافة من المنصة |

## 39. طرق الدفع المطلوبة في الاختبار

### الدفع عند الاستلام COD

1. فعّل طريقة الدفع عند الاستلام.
2. تأكد أنها تظهر للعميل.
3. نفذ طلبًا بها.
4. تحقق أن checkout لا يطلب إيصالًا أو رقم مرجع بلا داعي.

### كريمي أو محفظة محلية إن ظهرت

اختبر كريمي أو أي محفظة محلية ظاهرة كطريقة دفع من نوع `wallet`.

| الحقل | القيمة |
|---|---|
| اسم الحساب | Rose Oud |
| رقم الحساب أو المحفظة | 123456789 |
| رقم الهاتف | +9677xxxxxxx |
| التعليمات بالعربي | حول المبلغ ثم اكتب رقم العملية، وارفع الإيصال إذا توفر. |
| التعليمات بالإنجليزي | Transfer the amount, enter the reference number, and upload receipt if available. |

## 40. اختبارات مهمة للدفع

1. فعّل طريقة دفع واحدة على الأقل.
2. عطّل جميع الطرق مؤقتًا إن أمكن، ثم افتح checkout وتحقق أن الرسالة واضحة.
3. اختبر طريقة تحتاج رقم مرجع إن ظهر حقل المرجع.
4. اختبر إرسال الطلب بدون مرجع إذا كان المرجع مطلوبًا.
5. اختبر رفع إيصال تجريبي.
6. اختبر إرسال الطلب بدون إيصال.
7. إذا كان النظام يقول إن الإيصال مطلوب لكنه يسمح بالإرسال بدونه، سجلها كمشكلة عالية.
8. إذا كان الإيصال اختياريًا وظهر ذلك بوضوح، اعتبره سلوكًا صحيحًا.

# المرحلة الثانية عشرة: التوصيل والشحن والاستلام

## 42. اختبار مهم قبل الإعداد

قبل إنشاء مناطق أو طرق توصيل، افتح المتجر كعميل وحاول الوصول إلى checkout.

المطلوب تسجيل ما يحدث:

- هل يمنع النظام إكمال الطلب؟
- هل تظهر رسالة واضحة للعميل؟
- هل الرسالة تشرح أن المتجر لم يفعّل التوصيل؟
- هل يستطيع العميل اختيار الاستلام من المتجر إذا كان مفعّلًا؟

## 43. مناطق الشحن المطلوبة

| الاسم | المدينة/المحافظة | المنطقة | الرسوم الافتراضية | الحالة |
|---|---|---|---|---|
| صنعاء داخل المدينة | أمانة العاصمة | داخل المدينة | 1500 | فعالة |
| صنعاء أطراف | صنعاء | أطراف المدينة | 2500 | فعالة |
| عدن | عدن | داخل المدينة | 3500 | فعالة |

## 44. طرق الشحن/التوصيل المطلوبة

| النوع | اسم العرض | الحقول المهمة |
|---|---|---|
| flat_rate | توصيل ثابت داخل صنعاء | cost، minDeliveryDays، maxDeliveryDays |
| by_weight | توصيل حسب الوزن | baseCost، costPerKg، minCost، maxCost |
| by_item | توصيل حسب عدد القطع | baseCost، costPerItem |
| free_shipping | شحن مجاني للطلبات الكبيرة | freeShippingConditionType، freeShippingMinOrderAmount |
| store_pickup | استلام من المتجر | pickupCost، أيام التوصيل أو تعليمات الاستلام |
| weight_tier | شرائح حسب الوزن | ranges |
| order_value_tier | شرائح حسب قيمة الطلب | ranges |

## 45. خطوات التنفيذ

1. أنشئ مناطق التوصيل أعلاه.
2. أنشئ طريقة توصيل ثابتة داخل صنعاء.
3. أنشئ طريقة مجانية عند تجاوز مبلغ معين إن كانت متاحة.
4. أنشئ طريقة استلام من المتجر إن كانت متاحة.
5. أنشئ طريقة حسب الوزن أو الشرائح إن كانت متاحة.
6. افتح checkout واختبر تغير رسوم الشحن حسب المنطقة أو الطريقة.
7. نفذ طلبًا باستخدام التوصيل.
8. نفذ طلبًا ثانيًا باستخدام الاستلام من المتجر إن كان متاحًا.

## 46. تحقق المرحلة

- [ ] استطعت إنشاء منطقة توصيل.
- [ ] استطعت إنشاء أكثر من طريقة توصيل/شحن.
- [ ] فهمت معنى نوع كل طريقة.
- [ ] ظهرت طرق الشحن في checkout.
- [ ] تغيرت الرسوم حسب المنطقة أو الطريقة.

- [ ] ظهر الشحن المجاني فقط عند تحقق الشرط.

# المرحلة الخامسة عشرة: تتبع الطلب من جانب العميل

## 52. المطلوب الفعلي

صفحة تتبع الطلب في النظام الحالي مختصرة. لا تتوقع منها عرض كل تفاصيل الطلب مثل صفحة لوحة التاجر.

البيانات المتوقعة فعليًا:

- رقم الطلب.
- حالة الطلب.
- الإجمالي.
- العملة.
- آخر تحديث.
- خط زمني أو مراحل الحالة إن ظهرت.

> إذا لم تظهر المنتجات أو الدفع أو الشحن في صفحة التتبع، لا تعتبر ذلك فشلًا وظيفيًا مباشرًا. سجله كتوصية تحسين لتجربة العميل.

## 53. خطوات التنفيذ

1. افتح صفحة تتبع الطلب.
2. أدخل رقم الطلب أو البيانات المطلوبة.
3. تأكد من ظهور حالة الطلب.
4. تأكد من ظهور الإجمالي والعملة إن وجدا.
5. تأكد من ظهور آخر تحديث أو timeline إن وجد.
6. بعد تغيير حالة الطلب من لوحة التاجر، أعد فتح صفحة التتبع وتأكد من تحديث الحالة.
7. تحقق أن أسماء الحالات في صفحة التتبع مطابقة لحالات النظام الفعلية، وليست حالات قديمة مثل pending/processing/shipped/delivered.

## 54. تحقق المرحلة

- [ ] وجدت صفحة تتبع الطلب.
- [ ] عرفت ما البيانات المطلوبة للتتبع.
- [ ] ظهرت حالة الطلب.
- [ ] ظهر الإجمالي أو بيانات مختصرة مفيدة.
- [ ] تغيرت الحالة بعد تحديث التاجر.
- [ ] حالات التتبع مطابقة لحالات النظام الفعلية أو تم تسجيل الفرق كمشكلة.
- [ ] نقص تفاصيل المنتجات/الدفع/الشحن سجل كتوصية، وليس كفشل أساسي.

# المرحلة السابعة عشرة: تغيير حالة الطلب

## 57. الحالات الفعلية

| الحالة التقنية | التسمية العربية |
|---|---|
| new | جديد |
| confirmed | مؤكد |
| preparing | قيد التجهيز |
| out_for_delivery | في الطريق |
| completed | مكتمل |
| cancelled | ملغي |
| returned | مسترجع |

## 58. الانتقالات المسموحة

| من | إلى |
|---|---|
| new | confirmed أو cancelled |
| confirmed | preparing أو cancelled |
| preparing | out_for_delivery أو cancelled |
| out_for_delivery | completed أو returned |
| completed | لا يوجد انتقال |
| cancelled | لا يوجد انتقال |
| returned | لا يوجد انتقال |

## 59. خطوات التنفيذ

1. غيّر الطلب من `new` إلى `confirmed`.
2. أضف ملاحظة إن ظهر حقل الملاحظة.
3. غيّره إلى `preparing`.
4. غيّره إلى `out_for_delivery`.
5. افتح صفحة تتبع الطلب وتأكد من ظهور الحالة.
6. غيّره إلى `completed`.
7. أنشئ طلبًا ثانيًا أو استخدم طلب اختبار آخر لتجربة الإلغاء.
8. جرّب انتقالًا غير مسموح مثل `new` إلى `completed` إن أمكن، وسجل هل تم منعه.

## 60. تحقق المرحلة

- [ ] الحالات المعروضة مطابقة للقائمة الفعلية.
- [ ] لا توجد حالة "قيد المراجعة" كحالة طلب.
- [ ] الانتقالات غير المسموحة ممنوعة أو غير معروضة.
- [ ] الملاحظات تحفظ إن وجدت.
- [ ] حالة العميل في صفحة التتبع تتحدث.

# المرحلة الثامنة عشرة: الإشعارات

## 61. ما يجب فحصه

1. إشعار إنشاء طلب جديد.

3. إشعار نفاد المخزون عند وصول المخزون إلى 0.

   - فورية.

أحداث مهمة يجب تسجيلها إن ظهرت:

- `order.created`

## 62. تحقق المرحلة

- [ ] يمكن تمييز الإشعار كمقروء.
- [ ] تفضيلات الإشعارات واضحة إن ظهرت.
- [ ] الإشعارات الفورية تعمل أو لا تعمل وسجلت النتيجة.
